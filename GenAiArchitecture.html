My Architecture: RAG + Caching + Orchestration

The system is split into two phases — an offline ingestion pipeline (runs once) and an online query pipeline (runs every user request).

Retrieval pipeline Vector search (ANN) Semantic caching Orchestration layer Latency optimization Horizontal scalability

Phase 1 — Offline Ingestion Pipeline

Run once when setting up the system. Prepares all documents so they can be searched at lightning speed.

⚙️ One-time setup · runs offline
📁 Knowledge Base

Upload all your documents — product manuals, FAQs, policy files. This is the source of truth for the AI.

✂️ Document Chunking

Each document is split into small segments (~512 tokens). Smaller pieces = faster and more accurate search results.

🔢 Embedding Model

Each chunk is converted into a vector (list of numbers) that represents its meaning. Similar text = similar numbers.

🗄️ Vector Database

All vectors are stored here. It can search millions of vectors by meaning in milliseconds using ANN algorithms.

Phase 2 — Online Query Pipeline

Runs every time a user asks a question. Optimized for speed (caching) and accuracy (retrieval + re-ranking).

⚡ Real-time · every user request
✅ Return answer to user
⚡ Step A — Cache Check

First check if the same (or similar) question was answered before. If yes, return instantly — no AI needed. Redis handles this.

🔢 Step B — Embed + Search

Convert the query to a vector. Run ANN search in the Vector DB to find the top-5 most relevant document chunks by meaning.

📊 Step C — Re-rank

A cross-encoder re-orders the retrieved chunks. The most relevant chunk moves to top. Improves accuracy significantly.

📝 Step D — Build Prompt

Combine: System instructions + Retrieved chunks (context) + User question → one structured prompt sent to the LLM.

🤖 Step E — Orchestrate LLM

The orchestrator classifies query complexity. Simple → Gemini Flash (fast, cheap). Complex → Gemini Pro (accurate, powerful).

💾 Step F — Cache + Return

Save the answer in cache (with TTL to avoid stale data). Return the answer to the user. Next similar question = instant reply.

Trade-offs & Design Decisions

Every design choice has a trade-off. Here are the key ones and how this architecture handles them.

⚡

Cache speed vs. freshness

Cache improves speed but may return outdated answers if knowledge base is updated.

→ Solution: Set TTL (Time-To-Live) on cache entries. Expire old answers automatically.
✂️

Chunk size vs. accuracy

Very small chunks lose context. Very large chunks make the LLM read too much irrelevant text.

→ Solution: 512 tokens per chunk is the proven sweet spot for most use cases.
🤖

Gemini Flash vs. Pro routing

Using Pro for everything = expensive and slower. Flash for everything = less accurate on hard queries.

→ Solution: Orchestrator classifies query complexity and routes to the right model.
🔍

ANN (approximate) vs. exact search

Exact nearest neighbor search is too slow at scale. ANN is slightly approximate but 100x faster.

→ Solution: ANN is accurate enough (>99% recall) and returns results in milliseconds.

Scalability Design

How the architecture handles high traffic without slowing down.

Vector DB Scaling

Vertex AI Vector Search scales horizontally. Handles billions of vectors automatically.

Cache Layer

Redis Cluster absorbs repeated query traffic. Most common questions never reach the LLM.

LLM API

Vertex AI manages LLM scaling automatically. Use async and streaming to reduce wait time.

Serving Layer

Cloud Run auto-scales the API from 0 to thousands of instances based on traffic demand.
